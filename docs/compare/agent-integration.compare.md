# Agent Integration (Claude / Copilot / Kiro) — Comparison

> **Proposed verdict:** **Both — instruction files *and* hooks, generated from one
> source by `fux setup --agents`.** Canonical instructions live in **AGENTS.md** (the
> open standard read natively by Copilot, Cursor, Codex, Windsurf, Zed — and picked up
> by Kiro and Claude Code); `fux setup --agents` writes it plus thin tool-native pointers
> (`CLAUDE.md` section, `.github/copilot-instructions.md`,
> `.kiro/steering/fux.md`) so every agent surface — CLI, VS Code extension, Kiro IDE —
> gets the same contract. Hooks are the *enforcement* layer where supported: a Claude
> Code `UserPromptSubmit` hook and a Kiro agent hook that run `fux ask --json` and
> inject results, so querying isn't left to the agent's memory. The "something
> better" — an MCP server — is real but **deferred** (explicitly out of scope in
> CLAUDE.md without an ADR + sign-off).
> **Status:** ✅ Accepted (Arpit, 2026-07-21) · **Confidence:** Medium-High
> Extension accepted same day: **one skill, not three** — a single `SKILL.md` in the
> Agent Skills open standard serves Claude, Copilot, and Kiro alike (see "Skills"
> section below).

## Context

Fux's whole point is that agents *consult the knowledge before acting* — so agents
need to (a) know fux exists, (b) know when to run `fux ask` / `find` / `answer`, and
(c) ideally have that happen automatically rather than by good intentions. Arpit's
requirement: works with **Claude (Code + Cowork), GitHub Copilot (CLI and VS Code),
and Kiro**, via hooks, md-file additions, or both — and evaluate whether something
better exists.

## What each tool actually reads (researched, July 2026)

| Surface | Instruction file | Hook mechanism |
|---------|-----------------|----------------|
| Claude Code / Cowork | `CLAUDE.md` (native, richest); AGENTS.md support present | Hooks incl. `UserPromptSubmit` — run a command on every prompt, inject output |
| GitHub Copilot (VS Code + CLI) | `.github/copilot-instructions.md`; reads **AGENTS.md** natively | No first-class local hook for injection |
| Kiro (IDE + CLI) | `.kiro/steering/*.md`; picks up **AGENTS.md** automatically (workspace root or `~/.kiro/steering/`) | Agent hooks in `.kiro/hooks/` — event-driven, natural-language automations |
| Cursor / Codex / Windsurf / Zed (bonus) | **AGENTS.md** natively | varies |

AGENTS.md is the convergence point: an open standard (originated by OpenAI, now under
the Linux Foundation's Agentic AI Foundation, 60k+ repos) that nearly every agent
reads — which is why it's the canonical copy and the tool-native files are pointers.

## Options

- **A — Instruction files only.** AGENTS.md + tool-native pointers tell agents when to
  run fux. Zero moving parts; works everywhere including Copilot. But it's advisory —
  an agent can forget.
- **B — Hooks only.** Claude Code `UserPromptSubmit` + Kiro hooks auto-run fux and
  inject results. Enforced, not advisory — but Copilot has no equivalent hook, so
  Copilot users get nothing.
- **C — Both, generated from one source.** *(proposed)* `fux setup --agents` scaffolds
  the files; `fux init-agents --hooks` adds the hook wiring where the tool supports
  it. One template, one command, no drift between copies.
- **D — MCP server.** The architecturally "better" answer — a first-class tool the
  agent can call with structured results. Deferred: out of scope per CLAUDE.md
  (revival needs an ADR); heavier to maintain; and file+hook coverage already reaches
  all three required tools today.

## The generated contract (what the files say)

One template, rendered into each surface, teaching the agent:

1. **Before answering questions about this project's docs/decisions**, run
   `fux ask "<question>" --json` and prefer its passages over guessing.
2. `fux find` to locate documents; `fux answer --explain` for a cited extractive
   answer plus reasoning.
3. If a cited source shows `fidelity: inferred` and looks thin, run
   `fux ingest --advanced <file>` and re-ask.
4. After ingesting new folders, re-run your question — the index updates.

The hook layer (where supported) runs step 1 automatically on each user prompt and
injects the top passages, making consultation structural instead of remembered — the
same instinct as the held rule engine's "read before you touch."

**Doc-freshness prompting (accepted 2026-07-21).** A second hook closes the loop in
the other direction: at session end (Claude Code `Stop` hook / Kiro on-save hook), it
diffs the session's changed files against the trigger column of
[`../DOC-REGISTRY.md`](../DOC-REGISTRY.md) and *prompts*: "these docs look affected —
update them and bump their registry rows?" Advisory and fail-open — it nags, never
blocks. The generated AGENTS.md/instruction files carry the same rule as prose (step
5 of the contract: "before finishing, check DOC-REGISTRY.md triggers against your
changes"), so even hook-less tools (Copilot) get the prompt. Research grounding:
docs-as-code practice is doc-updates-in-the-same-change with freshness signals
(last-reviewed dates, owners, CI freshness scoring) — the registry is that signal
made local and checkable.

## Skills — one SKILL.md for all three tools (accepted 2026-07-21)

Arpit asked whether to create skills for Copilot, Claude, and Kiro. Research says:
**yes to skills, no to per-tool skills.** In December 2025 Anthropic released **Agent
Skills as an open standard** (spec at agentskills.io), and by March 2026 **32+ tools
read the same `SKILL.md` from the same directory structure** — including GitHub
Copilot, AWS Kiro, OpenAI Codex, Cursor, Gemini CLI, and VS Code. A well-written
skill works across all of them without modification.

This is a meaningful simplification over the old build, whose `tools/skillgen`
rendered *per-platform* skill variants from shared fragments precisely because no
standard existed. Now the platforms converged on Anthropic's format — one file, no
generator needed for the skill itself.

**Where skills fit in the layer cake** (each layer does a different job):

| Layer | File | Job |
|-------|------|-----|
| Instructions | `AGENTS.md` (+ pointers) | *When* to use fux — always-loaded project contract |
| Skills | `skills/fux/SKILL.md` | *How* to run the workflows — loaded on demand when relevant |
| Hooks | Claude Code / Kiro | *Enforcement* — auto-inject, no memory required |

Two skills ship with the package, in the standard format:
- **`fux-query`** — the ask→judge→explain workflow: run `fux ask --json`, evaluate
  passage quality, escalate to `fux answer --explain`, cite sources in replies.
- **`fux-ingest`** — the ingest workflow already specified in
  [`ingest-strategy.compare.md`](ingest-strategy.compare.md): inferred ingest, read
  `--list-inferred`, judge extraction quality, trigger `--advanced`, `--check` drift.

`fux setup --agents` places them (or symlinks) into each tool's skill discovery path;
the skill *content* is written once.

## Additional things to look into

- **`setup --agents` idempotency:** re-running must update the fux-managed block without
  clobbering user content (marked begin/end comments).
- **Hook safety:** the Claude Code hook must be fail-open (a fux error never breaks
  the session) — the same hook contract as the old build.
- **Copilot gap:** watch for Copilot gaining a hook/injection mechanism; until then
  Copilot relies on the instruction files alone.
- **MCP revival trigger:** if agents misuse the CLI contract in practice, that's the
  signal to write the MCP ADR rather than patching prose.
