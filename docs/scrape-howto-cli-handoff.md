# Fux — Web-Rules, Self-Docs, and CLI Help Handoff

**Owner:** Arpit · **Repo:** `fux` · **Driving model:** Claude Code.
**Scope:** four related features — (A) scrape websites → draft rules, (B) configurable CDP port, (C) `fux how` self-documentation, (D) better CLI help. C and D share one data source, so build D first.

---

## 0. Non-negotiables (fux's identity — do not break)

- **`$0`, stdlib only, deterministic** on the maintenance path. No network and no LLM in `check`/`gate`/`verify`/`seal`/`recall`/`how`. A guard test must prove it.
- **The agent fetches and interprets; fux governs.** Network (HTTP/CDP) and turning text into rule prose are the *host agent's* tokens, via a skill — never the engine. Same fence as `/fux debate` and `/fux critic`.
- **Web rules are low-trust:** drafted as `status: draft`, never auto-active, never auto-constitutional.
- Files ≤ 100 lines (≤ 50 for `*_utils`). No new runtime deps. Docs in the same change (`fux-plan.md`, `cli.md`, `README.md`).

## A. Scrape websites → draft rules (`/fux scrape <url>`)

A skill, not engine code. The agent does the fetching and extraction; fux validates and governs the result.

1. **Fetch (agent-side, CDP escalation).** Try a plain HTTP fetch. If the page is a client-rendered shell (spinner / "enable JavaScript" / empty body), escalate to **CDP** (render, then read text). The skill chooses the backend; the engine never fetches.
2. **Classify → type + trust:** docs/API → `type: convention`; own/team docs → `type: rule`/`glossary`; market/data → `type: rule` (data caveat); **regulatory/tax/compliance → `type: regulatory`, DRAFT-VERIFY** (a scrape is never the authority — verify against the primary source, human ratify mandatory).
3. **Draft with provenance.** Agent writes Rule/Why/Edge as `status: draft` with new optional frontmatter: `source: <url>`, `fetched: <ISO date>`, `source_hash: <hash of extracted content>`.
4. **Govern.** `fux check` (schema) → review → `/fux debate` → `fux ratify` for anything that binds. Never auto-active/constitutional.
5. **Opt-in re-verify (fenced, network → an extra).** `fux scrape --recheck` re-fetches a rule's `source`, recomputes `source_hash`, raises `source-drift` if the page changed since `fetched`. Never on the default `fux check` path.

**Ships:** the `scrape` skill doc; schema fields `source`/`fetched`/`source_hash`; `--recheck` behind an opt-in extra; the `$0`-guard test extended to prove `scrape`'s engine side is network/LLM-free.

## B. Configurable CDP port

The CDP backend must not hard-code `:9299`. Resolve the CDP endpoint in this precedence: `--cdp-port` / `--cdp-host` flags → `FUX_CDP_PORT` / `FUX_CDP_HOST` env → `cdp_port` / `cdp_host` in `.fux/config.toml` → default `127.0.0.1:9299`. One small resolver (`≤50` lines, utils-style) shared by the scrape skill's fetch step. Document the precedence in `cli.md`.

## C. fux explains fux — `fux how` + self-scoped graph queries

fux's best dogfood subject is fux. Ship a **self-knowledge bundle** generated deterministically from fux's own source — its module graph (AST extraction over `fux/*.py`, the same engine), fux's own `.fux/rules`, and the bundled docs + command registry from (D). Built `$0` / AST-only at package time, shipped in `data/self/`. It answers "how does fux *itself* work," not just the project documents fux holds. Two surfaces query it:

- **`fux how "<question>"`** — quick answer **+ the exact command**, via deterministic recall (BM25F, `$0`) over the registry + self-docs. `fux how "how does drift detection work"` → explanation + `fux seal` / `fux check`; `fux how "which rules govern a file"` → `fux refs <path>`.
- **`fux query --self` / `explain --self` / `path --self`** — graph traversal over fux's *own architecture*: the same commands that traverse a user's project, pointed at the fux bundle. `fux explain --self "drift"` → how `seal`/`check`/`drift` relate across fux's real modules (EXTRACTED + INFERRED edges), grounded in code, not prose. **This is what answers "how does fux work" properly** — from fux's actual code relationships.

Default scope is the user's project; **`--self` targets the shipped fux bundle and works in any repo** (even one with no `.fux/`). Deterministic by default; the optional `--explain` (host-agent tokens, fenced like the critic) is the only model path. The bundle is regenerated by `fux self-build` (AST-only, `$0`) whenever fux's source changes — so fux's self-knowledge can't drift from fux's code.

**Ships:** the self-bundle generator (`fux self-build`) + bundled `data/self/`; the `how` command (`howto.py` ≤100) over the registry+docs corpus; a `--self` scope on `query`/`explain`/`path`/`recall`; `--explain` opt-in.

## D. Improve the CLI help (build this first — it's C's corpus)

Make help discoverable *and* machine-usable, from **one source of truth**:

- A single structured **command registry** — each command: `name`, `group` (authoring · verification · governance · runtime), one-line `description`, and a copy-paste `example`. Both `fux --help` and `fux how` (C) consume this exact table.
- `fux --help` groups commands by `group` with aligned descriptions (not a flat dump).
- `fux help <command>` (and `fux <command> --help`) prints the description, usage, the example, and related commands.
- Keep it stdlib (argparse or the existing dispatcher); no new deps.

**Ships:** the command registry (single source of truth), grouped top-level help, per-command help, and `cli.md` regenerated from the registry so docs can't drift from help.

## Acceptance

- `fux --help` is grouped and readable; `fux help <cmd>` shows description + example; `cli.md` is generated from the same registry (no drift).
- `fux how "<question>"` returns a correct command + explanation deterministically for a known question set; `--explain` is the only path that uses a model.
- `fux query/explain/path --self "<question>"` answers from fux's **own module graph** (e.g. how `seal`/`check`/`drift` relate) in any repo, with no project `.fux/` required; the `data/self/` bundle regenerates byte-identically from fux's source.
- CDP port/host is configurable by flag, env, and config, in that precedence; default `9299`.
- Scrape produces `status: draft` rules with `source`/`fetched`/`source_hash`; regulatory drafts are flagged verify-against-source; nothing auto-ratifies.
- `$0`/stdlib/determinism intact; guard test green; files ≤100 lines; docs in sync.
